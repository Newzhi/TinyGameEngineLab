# 第一个三角形与顶点数据传递（基于当前 `main.cpp`）

这份文档对应 Day 02 的 `Proj/src/main.cpp`。Day 01 你已经能「开窗 + 清屏」，Day 02 的目标是：

1. 理解 **Shader、VBO、VAO** 各自是什么
2. 搞清楚 **顶点数据从 CPU 到 GPU 再到屏幕** 的完整路径
3. 知道常用 API 在做什么、为什么要按这个顺序写

---

## 1. Day 01 → Day 02：多了什么？

Day 01 的渲染循环只做「擦黑板」：

```
清屏 → 交换缓冲 → 轮询事件
```

Day 02 在清屏之后，多了一步「真正画画」：

```
清屏 → 启用 Shader → 绑定 VAO → glDrawArrays 画三角形 → 交换缓冲 → 轮询事件
```

同时，在程序启动阶段（进入渲染循环之前），还要做两件一次性工作：

- **编译链接着色器程序**（告诉 GPU 怎么算顶点、怎么上色）
- **创建 VAO / VBO，把顶点数据上传到 GPU**（告诉 GPU 画什么形状）

---

## 2. 核心概念速览

| 概念 | 全称 | 一句话理解 |
|------|------|-----------|
| **Shader** | 着色器 | 跑在 GPU 上的小程序：顶点着色器处理「点」，片段着色器处理「像素颜色」 |
| **VBO** | Vertex Buffer Object | 一块 GPU 显存，专门存顶点数据（坐标、颜色、UV 等） |
| **VAO** | Vertex Array Object | 记录「从哪块 VBO 读、怎么解读每个顶点」的配置快照 |
| **NDC** | Normalized Device Coordinates | 标准化设备坐标，范围大约 `[-1, 1]`，`(0,0)` 在屏幕中心 |
| **Draw Call** | 绘制调用 | 如 `glDrawArrays`，让 GPU 根据已有数据真正开始 rasterize |

可以把它想成：

- **VBO** = 原材料仓库（顶点坐标存在这里）
- **VAO** = 加工说明书（告诉 GPU 怎么读仓库里的 float）
- **Shader** = 加工流水线（顶点怎么变换、像素什么颜色）
- **glDrawArrays** = 按下「开始生产」按钮

---

## 3. 顶点数据是怎么传递的？（完整链路）

这是 Day 02 最重要的部分。数据一共经历 **4 段旅程**：

```
① CPU 内存          ② GPU 显存 (VBO)       ③ 顶点着色器           ④ 片段着色器 → 屏幕
vertices[]    →    glBufferData 上传   →   读 aPos，输出位置   →   上色 → 帧缓冲
```

### 3.1 第一段：CPU 上定义顶点

```cpp
float vertices[] = {
    -0.5f, -0.5f, 0.0f,   // 顶点 0：左下
     0.5f, -0.5f, 0.0f,   // 顶点 1：右下
     0.0f,  0.5f, 0.0f    // 顶点 2：顶部
};
```

这里每个顶点 **3 个 float**（x, y, z），三个顶点组成一个三角形。

坐标用的是 **NDC**：

- x、y 在 `[-1, 1]` 之间，直接对应屏幕可见区域
- 不需要 Day 01 那种 `glViewport` 之外的额外变换（初学阶段）
- z 这里都是 `0.0`，表示在屏幕平面上

> 注意：`vertices[]` 只是普通 C++ 数组，还在 **CPU 内存** 里。此时 GPU 完全不知道这些数据的存在。

### 3.2 第二段：上传到 GPU（VBO）

```cpp
unsigned int VBO;
glGenBuffers(1, &VBO);                              // 申请一个 buffer 对象 id
glBindBuffer(GL_ARRAY_BUFFER, VBO);                  // 绑定到 GL_ARRAY_BUFFER 目标
glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);
```

三步分别做什么：

| API | 作用 |
|-----|------|
| `glGenBuffers` | 向 OpenGL 申请一个 buffer 的「编号」（id），此时还没有实际显存 |
| `glBindBuffer` | 把编号为 `VBO` 的 buffer **绑定到当前目标**（这里是 `GL_ARRAY_BUFFER`，即顶点数组缓冲） |
| `glBufferData` | 分配显存，并把 `vertices` 的内容 **拷贝进 GPU** |

`glBufferData` 最后一个参数 `GL_STATIC_DRAW` 的含义：

- **STATIC**：数据基本不会改（适合这个固定三角形）
- **DRAW**：数据用来绘制（不是只读、不是下载回 CPU）

其他常见 hint（后面会用到）：

- `GL_DYNAMIC_DRAW`：数据会频繁更新（比如每帧改顶点）
- `GL_STREAM_DRAW`：数据每帧都换（比如粒子系统）

> 关键：`glBufferData` 之前 **必须先** `glBindBuffer`。OpenGL 的很多操作都是「对当前绑定的对象生效」。

### 3.3 第三段：告诉 GPU 怎么解读 VBO（VAO + 顶点属性）

光有 VBO 还不够。VBO 里只是一串连续的 float：

```
-0.5, -0.5, 0.0,  0.5, -0.5, 0.0,  0.0, 0.5, 0.0
```

GPU 不知道「每几个 float 是一个顶点、分别代表什么」。所以需要 **顶点属性配置**：

```cpp
glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, 3 * sizeof(float), (void*)0);
glEnableVertexAttribArray(0);
```

配合顶点着色器里的：

```glsl
layout (location = 0) in vec3 aPos;
```

逐项解释 `glVertexAttribPointer`：

| 参数 | 你的代码 | 含义 |
|------|---------|------|
| `index` | `0` | 对应 shader 里 `layout(location = 0)`，即 `aPos` 这个输入 |
| `size` | `3` | 每个顶点属性有 3 个分量（x, y, z） |
| `type` | `GL_FLOAT` | 每个分量是 float |
| `normalized` | `GL_FALSE` | 不做 `[0,255] → [0,1]` 之类的归一化 |
| `stride` | `3 * sizeof(float)` | 相邻两个顶点之间间隔 12 字节（3 个 float） |
| `offset` | `(void*)0` | 从 buffer 开头读起 |

`glEnableVertexAttribArray(0)` 则是 **打开** 这个属性通道。不 enable，shader 里的 `aPos` 读不到 VBO 数据。

**VAO 的作用**：上面这些配置（绑了哪个 VBO、stride/offset 是多少、哪些 attribute 启用了）可以记录到 VAO 里：

```cpp
glGenVertexArrays(1, &VAO);
glBindVertexArray(VAO);          // 开始记录配置
glBindBuffer(GL_ARRAY_BUFFER, VBO);
glBufferData(...);
glVertexAttribPointer(0, ...);
glEnableVertexAttribArray(0);
glBindVertexArray(0);            // 停止记录（可选）
```

之后每帧绘制只需：

```cpp
glBindVertexArray(VAO);          // 一次性恢复所有顶点配置
```

而不必每帧重复 `glBindBuffer` + `glVertexAttribPointer`。

### 3.4 第四段：Shader 处理 + 绘制

**顶点着色器**（每个顶点执行一次）：

```glsl
layout (location = 0) in vec3 aPos;

void main()
{
    gl_Position = vec4(aPos.x, aPos.y, aPos.z, 1.0);
}
```

- 从 location 0 读到 `aPos`（就是 VBO 里传上来的 x,y,z）
- 输出 `gl_Position`：顶点的最终裁剪空间坐标

**片段着色器**（每个像素执行一次）：

```glsl
out vec4 FragColor;

void main()
{
    FragColor = vec4(1.0f, 0.5f, 0.2f, 1.0f);  // 橙色
}
```

三角形内部每个像素都会被赋这个颜色。

**Draw Call**：

```cpp
glUseProgram(shaderProgram);     // 启用着色器程序
glBindVertexArray(VAO);          // 绑定顶点配置
glDrawArrays(GL_TRIANGLES, 0, 3); // 从第 0 号顶点起，取 3 个顶点，拼成 1 个三角形
```

`glDrawArrays` 触发 GPU 流水线：

1. 读 VAO 关联的 VBO
2. 按配置解析出 3 个顶点
3. 顶点着色器处理 3 次
4. 光栅化：三角形内部生成像素
5. 片段着色器给每个像素上色
6. 写入帧缓冲（之后由 `glfwSwapBuffers` 显示）

## 4. Shader 编译链接流程

OpenGL 3.3 Core **没有固定管线**，不写 Shader 就什么都画不出来。

你的 `main.cpp` 里拆成了两个辅助函数，流程如下：

```
顶点 Shader 源码 ──→ glCreateShader + glCompileShader
片段 Shader 源码 ──→ glCreateShader + glCompileShader
         ↓
    glCreateProgram
    glAttachShader × 2
    glLinkProgram          ← 链接成一个可执行 Program
         ↓
    glDeleteShader × 2     ← 链接后 shader 对象可删
```

常用 API：

| API | 作用 |
|-----|------|
| `glCreateShader(type)` | 创建 shader 对象（`GL_VERTEX_SHADER` 或 `GL_FRAGMENT_SHADER`） |
| `glShaderSource` | 传入 GLSL 源码 |
| `glCompileShader` | 编译成 GPU 可执行代码 |
| `glCreateProgram` | 创建程序对象（最终用来 `glUseProgram` 的那个 id） |
| `glAttachShader` | 把编译好的 shader 挂到 program 上 |
| `glLinkProgram` | 链接：把 vertex + fragment 拼成完整流水线 |
| `glUseProgram` | 渲染时启用这个 program |
| `glGetShaderInfoLog` / `glGetProgramInfoLog` | 编译/链接失败时读错误日志 |

> `#version 330 core` 必须和你在 `glfwWindowHint` 里请求的 OpenGL 3.3 Core 一致。

## 5. Day 02 完整启动顺序（对照代码）

在 **GLAD 初始化成功之后**，按这个顺序做一次性 setup：

```
7) createShaderProgram(...)     // 编译链接
8) glGenVertexArrays / glGenBuffers
   glBindVertexArray(VAO)
   glBindBuffer + glBufferData   // 上传顶点
   glVertexAttribPointer + glEnableVertexAttribArray
```

渲染循环里每帧：

```
processInput
glClearColor + glClear          // 先擦黑板
glUseProgram
glBindVertexArray
glDrawArrays                    // 再画三角形
glfwSwapBuffers + glfwPollEvents
```

退出前：

```
glDeleteVertexArrays / glDeleteBuffers / glDeleteProgram
glfwTerminate
```

### 为什么 OpenGL 调用必须在 GLAD 之后？

Day 01 已经强调：**先有上下文 → 再 GLAD → 再调 OpenGL API**。

Shader 编译、`glGenBuffers`、`glBufferData` 全部是 OpenGL API。如果在 `glfwCreateWindow` / `gladLoadGLLoader` 之前调用，函数指针为空，会崩溃或行为未定义。

`vertices[]` 和 shader 源码字符串是 **纯 CPU 数据**，可以在窗口创建前就写好。

## 7. 常用 API 速查（Day 02 新增部分）

### 7.1 Buffer / VAO

- `glGenVertexArrays(n, &VAO)` — 生成 VAO id
- `glGenBuffers(n, &VBO)` — 生成 buffer id
- `glBindVertexArray(VAO)` — 绑定 VAO（后续顶点配置记在它上面）
- `glBindBuffer(GL_ARRAY_BUFFER, VBO)` — 绑定顶点 buffer
- `glBufferData(target, size, data, usage)` — 分配显存并上传数据
- `glVertexAttribPointer(index, size, type, normalized, stride, offset)` — 配置属性解读方式
- `glEnableVertexAttribArray(index)` — 启用某个顶点属性
- `glDeleteVertexArrays` / `glDeleteBuffers` — 释放 GPU 资源

### 7.2 Shader / Program

- `glCreateShader` / `glShaderSource` / `glCompileShader`
- `glCreateProgram` / `glAttachShader` / `glLinkProgram`
- `glUseProgram(shaderProgram)` — 渲染前启用
- `glDeleteProgram` / `glDeleteShader`

### 7.3 绘制

- `glDrawArrays(GL_TRIANGLES, first, count)` — 用当前 VAO + Shader 画 `count` 个顶点

## 8. 常见问题

| 现象 | 可能原因 |
|------|----------|
| 只有背景色，没有三角形 | 忘了 `glDrawArrays`；或忘了 `glUseProgram` / `glBindVertexArray` |
| 编译/链接报错 | GLSL 语法错误（如 `out vec4 FragColor` 缺分号）；`#version` 不匹配 |
| 程序崩溃 | 在 GLAD 初始化之前调用了 OpenGL API |
| 三角形位置/形状不对 | `vertices` 坐标写错；或 `glVertexAttribPointer` 的 size/stride 与 shader 不一致 |
| 拉伸窗口后三角形被压扁 | NDC 下正常现象；后面用投影矩阵（Day 04+）解决宽高比 |
| `glBufferData` 似乎没效果 | 忘记先 `glBindBuffer` |

## 10. 下一步建议（Day 03 方向）

当前三角形的 **颜色写死在片段着色器里**。接下来可以：

1. **画第二个三角形**：改顶点坐标，或复用同一套 VAO/VBO/Shader
2. **用 `glDrawArrays` 画多个图元**：理解 `first` 和 `count` 参数
3. **引入 EBO（Element Buffer Object）**：用索引复用顶点，画矩形只需 4 顶点 + 6 索引
4. **把颜色作为顶点属性**：每个顶点带 RGB，在顶点/片段着色器间插值（`layout(location = 1) in vec3 aColor`）

做到 EBO + 顶点颜色，你就从「一个死橙色三角形」进入「可复用、可着色的基础几何体」阶段了。
